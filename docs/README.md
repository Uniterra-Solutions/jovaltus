# Jovaltus 專案文檔

## 文檔結構

| 目錄                 | 內容                   | 撰寫規則                               |
| -------------------- | ---------------------- | -------------------------------------- |
| `docs/features/`     | 系統對使用者提供的功能 | BDD 格式，使用者視角，不引用程式碼     |
| `docs/architecture/` | 系統宏觀設計與模組邊界 | 設計原則，模組職責，邊界規則           |
| `docs/principles/`   | 程式碼撰寫與組織慣例   | 命名慣例、編碼風格、測試慣例、錯誤處理 |

`docs/jovaltus-proposal.md` 為原始產品提案與設計願景文件，不屬於上述三類結構化文檔，予以保留。

## 證據追溯

維護文檔時，每條 claim 必須附上來源檔案與行號區間。信心分數模型：

| 信心水準    | 分數 | 定義                           | 標記              |
| ----------- | ---- | ------------------------------ | ----------------- |
| Direct      | 0.9  | 可直接從程式碼結構驗證         | 無需標記          |
| Indirect    | 0.7  | 符合已知模式但無明確註冊       | 無需標記          |
| Inferred    | 0.5  | 根據命名慣例或部分結構比對推斷 | 標記 `[INFERRED]` |
| Speculative | 0.2  | 僅根據關鍵字或註解推測         | 省略不寫入文檔    |

無法從程式碼直接證明的 claim 標記為 `[INFERRED]` 而非偽造來源。

## LLM 安全原則

產生或更新文檔時，只餵給 LLM 結構化中繼資料，不傳輸原始碼完整內容。安全的中繼資料包括：

- 檔案列表與模組邊界
- API 端點路徑與 methods
- 函式簽名與型別定義（不含實作主體）
- 套件依賴關係（不含實際資料）

## 增量更新

當程式碼變更時，只重新產生受影響的文檔區段。使用以下流程：

1. 透過 `git diff` 識別變更範圍
2. 判斷哪些 `.md` 檔案受到影響
3. 僅修補受影響的章節，不全面重寫
4. 更新證據來源（檔案路徑 + 行號區間）

## 定期 Drift Detection

建議每月或每季比對文檔與實際程式碼，確認無重大偏離。發現 drift 時：

1. 執行 `apltk codegraph sync` 同步索引
2. 使用 `apltk codegraph status` 檢查索引覆蓋率
3. 比對文檔中的模組邊界與實際檔案結構
4. 只修補受影響章節，不全面重寫
