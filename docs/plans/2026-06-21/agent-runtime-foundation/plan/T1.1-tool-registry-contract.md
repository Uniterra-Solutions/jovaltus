### Background

Most tool functionality already exists. The known gap is that the registry currently retrieves by id only, while the spec requires selected tools by id or name. Existing tests also chain `.register(...).execute(...)`, but `register()` returns `void`, causing 17 test failures. Decide one contract and make implementation plus tests agree. The least disruptive contract is to keep `register(tool): void` and update tests to register on a local variable before executing.

## 3. Tasks

### `packages/core/src/tools/types.ts` — add name-based selection contract

1. Open `packages/core/src/tools/types.ts`.
2. Locate the `ToolRegistry` interface.
3. Keep `register(tool: Tool): void` unless there is a strong reason to make it chainable.
4. Add a lookup method that retrieves by tool display name without breaking existing id lookup. Preferred shape:
   - `getByName(name: string): Tool | undefined;`
5. Update comments so `get(id)` is clearly id-based and `getByName(name)` is clearly name-based.

### `packages/core/src/tools/registry.ts` — implement name-based lookup

1. Open `packages/core/src/tools/registry.ts`.
2. Add a second in-memory index for names, or derive lookup from the existing map. Preferred implementation:
   - Keep `tools = new Map<string, Tool>()`.
   - Add `toolsByName = new Map<string, Tool>()`.
   - On register, reject duplicate ids as currently implemented.
   - Also reject duplicate names with a clear error such as `Tool name 'Read' is already registered.`.
   - Store both `tool.definition.id` and `tool.definition.name`.
3. Add `getByName(name: string): Tool | undefined`.
4. Keep `list()` returning all registered tools.
5. Keep `execute()` id-based and structured-error-only.

### `packages/core/src/tools/registry.test.ts` — fix contract drift and cover requirements

1. Open `packages/core/src/tools/registry.test.ts`.
2. Replace every chain of this form:
   - `await createToolRegistry().register(createReadTool()).execute(...)`
3. With explicit local registry usage:
   - `const registry = createToolRegistry();`
   - `registry.register(createReadTool());`
   - `const result = await registry.execute(...)`
4. Add registry tests for:
   - retrieving a registered tool by name, for example `expect(r.getByName('Read')).toBe(tool)`.
   - returning `undefined` for an unknown name.
   - rejecting duplicate names if `registry.ts` implements duplicate-name protection.
5. Preserve existing tests for:
   - empty registry.
   - duplicate id rejection.
   - unknown tool structured error.
   - thrown handler structured `UNKNOWN_ERROR`.
   - read tool success and read-state marking.
   - read tool failures and invalid params.
   - write tool create, parent directory creation, unread existing file rejection, read existing file success, invalid params.
   - edit tool success, unread rejection, missing `oldString`, invalid params.
   - bash tool success, invalid params, and command failure.
6. Where tests create files, assert side effects when required by CHECKLIST:
   - after write success, read the file and assert content where useful.
   - after unread write/edit rejection, assert original content remains unchanged.
