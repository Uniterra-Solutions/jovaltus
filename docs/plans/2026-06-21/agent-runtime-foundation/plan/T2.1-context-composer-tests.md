## 1. Mission & Rules

### Mission

Add focused tests proving the context composer provides typed memory, skills, tool descriptions, and MCP context for a single-agent run without executing tools, while gracefully handling empty sources.

### Rules

- If a file or function cannot be found, or the implementation is blocked, report it immediately — do not guess or work around it.
- Do not add new dependencies.
- Workers are leaf nodes — do not spawn sub-workers.
- Do not change tool registry behavior in this task; T1.1 owns registry changes.

### Background

The context composer already exists, but there is no dedicated test file. The composer should return `memory`, `skills`, `tools`, and `mcps` sections. Default memory, skills, and MCP providers return `null`, while missing tool registry returns `{ tools: [] }`. When a registry is provided, tool context must read definitions and parameter schemas only; it must not execute tool handlers.

## 3. Tasks

### `packages/core/src/context/composer.test.ts` — add context tests

1. Create `packages/core/src/context/composer.test.ts`.
2. Import `describe`, `expect`, and `it` from `vitest`.
3. Import `createContextComposer` from `./composer.js`.
4. Import `createToolRegistry` from `../tools/registry.js` and `createReadTool` from `../tools/read-tool.js`.
5. Add a test named `returns empty default context when no providers or registry are configured`:
   - Create composer with no providers.
   - Call `compose()` with no options.
   - Assert:
     - `context.memory` is `null`.
     - `context.skills` is `null`.
     - `context.mcps` is `null`.
     - `context.tools.tools` equals `[]`.
6. Add a test named `uses custom providers and describes registered tools`:
   - Create custom memory, skills, and MCP providers returning non-empty payloads matching `ContextProvider` types.
   - Create a tool registry and register `createReadTool()`.
   - Create composer with custom providers.
   - Call `compose({ toolRegistry: registry })`.
   - Assert memory, skills, and MCP payloads match provider outputs.
   - Assert one tool descriptor exists with:
     - `id` equal to `file_read`.
     - `name` equal to `Read`.
     - a non-empty description.
     - `parameters.type` equal to `object`.
7. Add a test named `does not execute tool handlers while composing tool descriptions`:
   - Create a minimal fake registry object with `list()` returning one tool whose `handler` increments a local counter or throws if executed.
   - Call `compose({ toolRegistry: fakeRegistry })`.
   - Assert returned tool descriptor includes the fake tool metadata.
   - Assert the handler execution counter remains `0`.
8. Do not modify `composer.ts` unless a test reveals it cannot satisfy the spec. If a change is needed, keep it narrowly scoped.

### Optional `packages/core/src/context/types.ts` refinement

Only if TypeScript makes tests awkward because `ContextComposerOptions.toolRegistry` uses `any`, refine the type to a minimal structural type that does not import the full registry implementation:

```ts
toolRegistry?: {
  list(): readonly {
    readonly definition: {
      readonly id: string;
      readonly name: string;
      readonly description: string;
    };
    readonly parameters: ToolParameterSchema;
  }[];
};
```

Do not broaden the public context API.
